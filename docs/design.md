## Problem Statement

The builder is a CS undergraduate specializing in AI who needs a portfolio flagship
that lands an Applied AI / GenAI engineering job. Their prior project, REVO (a Chrome
extension that explains GitHub repos), is dead: "explain a repo" is now a free feature
of Cursor/ChatGPT/Copilot, and 2026 hiring research flags "repo summarizer" as a
portfolio red flag. The need is a project that demonstrates the three signals applied-AI
reviewers actually reward and that bootcamp portfolios lack: (1) the AI does something a
thin wrapper cannot, (2) correctness is measured (evals), (3) it ships as a deployed,
demoable thing with cost/latency awareness.

The chosen domain is AI security, specifically **prompt injection** — tricking an AI by
feeding it malicious instructions. The chosen wedge is **indirect (hidden-content)
prompt injection**: instructions hidden inside a web page or document that an AI reads
and obeys, which the human user never sees.

## What Makes This Cool

- The headline demo is visceral: a visitor watches a normal AI assistant summarize a web
  page and get hijacked by white-on-white text the human can't see — leaking a planted
  secret or injecting a scam link. People feel that "whoa."
- It is the fresh corner of a hot field. Everyone benchmarks *direct* attacks (the user
  types the trick). Hidden-content injection is newer, scarier, and less saturated for a
  solo builder, while still being a top industry concern (OWASP's #1 LLM risk).
- It does both halves: it *attacks* (finds the holes) and *defends* (proves a fix moves
  the number). Shipping a measured defense is the rarest, highest-signal thing a student
  portfolio can show.
- It reuses the one genuinely transferable skill REVO taught: fetching and processing web
  pages. The "AI reads a page" victim is REVO's machinery pointed at a new purpose.

## Constraints

- Solo builder, beginner-to-intermediate, ~4 weeks of part-time work for v1 (the 5-phase
  plan below is a weekend-per-phase minimum; the judge hand-labeling step can be trimmed if
  time runs short).
- Strong in TypeScript/React/Vite/Vercel (from REVO); not yet fluent in Python ML tooling.
- Small budget for model API calls — must use cheap model tiers and cache aggressively.
- Windows dev environment.
- Reuse REVO's deploy pipeline (Vercel), UI scaffold (React + Tailwind), and its
  web-content fetching approach (HTML parsing plus rate-limited concurrent requests,
  e.g. Cheerio + p-queue).
- Defensive/ethical scope only: test models the builder has API access to, plus built-in
  practice victims. Never attack third-party production apps without permission.

## Premises

All four were presented in office hours and accepted by the builder:

1. **The scanner is a commodity.** promptfoo (18k+ stars, the de facto open-source
   standard), garak (NVIDIA), and PyRIT (Microsoft) already auto-generate thousands of
   injection attacks. Building a from-scratch scanner means rebuilding promptfoo, worse —
   a red flag, not a portfolio win.
2. **We build ON the established tools and public benchmarks** (BIPIA for indirect
   injection, AgentDojo, InjecAgent) rather than from scratch. Knowing and composing the
   real 2026 ecosystem is itself the senior hiring signal.
3. **A leaderboard is only impressive if it adds something** beyond re-running a public
   benchmark: novel attacks, a rigorous reproducible method, or a measured defense.
4. **The single highest-signal artifact is shipping a defense and proving it moves the
   number** (attack success X% -> Y%) on a public corpus. That is measurement + security +
   shipping an outcome in one.

## Approaches Considered

### Approach A: The Broad Leaderboard (minimal / most shareable)
A deployed web leaderboard that runs a fixed attack suite against GPT, Claude, Llama, and
Gemini and ranks their robustness, with a transparent scoring method and a validated AI
judge. Effort M-L (~3 wks). Risk Med: thin moat ("ran someone's benchmark") unless novel
attacks are added; API cost across 4+ models. Pros: shareable artifact, clear "I measure
things" signal. Cons: a lab could out-build it in a weekend. Reuses REVO UI + serverless.

### Approach B: Prove a Defense (ideal / highest ceiling)
Build a prompt-injection shield (input sanitizing + a detector + hardened prompting), then
measure attack success on a public benchmark before vs. after the shield. Headline: "cut
successful attacks from X% to Y%." Effort L (~4-5 wks). Risk Med-High: must actually move
the number; beating existing defenses is not guaranteed and a weak result undercuts the
finish. Pros: highest hire signal; ships an outcome. Cons: hardest; risk of a weak result.

### Approach C: The Invisible-Trap Wedge (creative / RECOMMENDED — chosen)
Go narrow and deep on indirect (hidden-content) injection. Build (1) an interactive
playground where an AI summarizes a page and gets hijacked by invisible text, (2) a small
reproducible leaderboard scoped to this one attack type across models, and (3) a measured
defense (HTML/Unicode sanitizing + content/instruction separation) showing the attack rate
drop. Effort M-L (~4 wks part-time). Risk Med, lowered by the narrow scope. Pros: most
original; best demo; delivers a focused leaderboard AND a proven defense; reuses REVO's
web-fetching DNA. Cons: feels narrower than an all-attacks, all-models leaderboard.

## Recommended Approach

**Approach C.** It is the version a solo beginner can finish and make excellent, it is the
most original (the fresh corner of a hot field), and it quietly delivers all three wins at
once: a shareable leaderboard (scoped to hidden-content injection), the jaw-dropping live
demo, and the "I proved a defense works" resume line. It is also the only approach that
reuses the transferable part of REVO (fetching and processing web content).

### Components

1. **The Playground (the demo).** A web UI: pick a built-in page or paste a URL/markdown.
   An AI assistant (wired to a model API) performs a task (summarize). A toggle injects a
   hidden instruction (white-on-white text, HTML comment, markdown image alt-text, unicode
   tag smuggling, "instruction-in-data"). Show clean vs. poisoned side by side and watch
   the AI leak a planted canary secret or inject a scam link. This is the headline moment.

2. **The Attack Corpus (the dataset artifact).** A curated, categorized JSON set of
   indirect-injection payloads, seeded from public sources (BIPIA and similar, cited) plus
   the builder's own variations. Tagged by technique — hidden-text (white-on-white CSS),
   HTML-comment, markdown-trick (e.g. malicious image alt-text), unicode-smuggling, and
   instruction-in-data (a malicious instruction sitting in a data field the AI reads — e.g.
   a JSON, CSV, YAML, or form field like `name: "Ignore previous instructions and ..."`) —
   and by goal (instruction
   override, canary exfiltration, link injection). Published in the repo.

3. **The Harness + Judge (the eval rigor).** Runs each payload through each model adapter,
   collects responses, and decides success via three layers: canary-token leak
   (deterministic, zero false positives), rule/keyword checks, and an LLM-as-judge for
   fuzzy cases. The judge must itself be injection-resistant: use a cheap reliable model
   (e.g. GPT-4o-mini or Claude Haiku) with a hardened system prompt that explicitly refuses
   to follow any instruction found in the content it is grading, and feed it HTML-stripped,
   non-rendered input. The judge's accuracy is validated against a hand-labeled set of at
   least 50 samples, stratified across attack techniques and models, and reported. Tracks
   latency and cost per run.

4. **The Leaderboard (the headline, scoped).** Robustness score per model ("blocked X% of
   hidden-content attacks"), with a methodology page so it is reproducible. Cheap model
   tiers only (e.g., Groq Llama, GPT-4o-mini, Claude Haiku, Gemini Flash).

5. **The Defense + before/after (the killer resume line).** A mitigation module applied
   before the model reads content. The pipeline runs in order: (1) unicode normalization —
   apply NFKC form, strip zero-width characters (U+200B-U+200D, U+FEFF), and remove tag
   characters (U+E0000-U+E007F) used for invisible smuggling; (2) HTML/markdown
   sanitization — strip hidden text, comments, and styling tricks; (3) input tagging — wrap
   the untrusted content in explicit delimiters (e.g. `<untrusted>...</untrusted>`) with a
   system prompt instructing the model to treat anything inside purely as data, never as
   instructions (this is what "content/instruction separation" / "spotlighting" means). If
   a payload itself contains the delimiter token, the sanitizer escapes or strips it before
   wrapping so the attack cannot break out. Measure attack success with the defense ON vs.
   OFF on the same corpus -> "cut successful hidden-content attacks from X% to Y%." Ship the
   defense as a small reusable module.

### Stack
TypeScript end to end: Next.js (or Vite React) front end, Vercel serverless functions for
the harness, official SDKs for OpenAI / Anthropic / Groq (Llama) / Google. Reuses REVO's
web-content fetching, serverless backend, and React/Tailwind UI. The Python red-team tools
(garak/promptfoo) are referenced only for attack taxonomy in v1; running them to
cross-check results is deferred to future work to avoid Python setup burden.

## Open Questions

- **Spend cap:** a full run is roughly 3-4 models x ~30-50 payloads x defense ON/OFF, plus
  one judge call per response — on the order of 500-800 API calls. On cheap tiers that is
  about $50-100. Confirm a hard cap (~$100 for v1); cache every result (local JSON file in
  dev, Vercel KV in prod, keyed by model + payload id + defense on/off); and pre-compute the
  leaderboard so visitor traffic never re-bills.
- **Corpus sourcing & licensing:** confirm BIPIA (and any other public payloads) license
  compatibility before reuse and cite them in the corpus JSON; if licensing is unclear for
  v1, author 30-50 original payloads only and defer reuse to v2.
- **Live playground abuse control:** pre-compute the demo cases; put the "run it live"
  button behind a per-IP rate limit (e.g. ~5 runs/hour) to cap cost and abuse.
- **Model set:** confirm the exact cheap-tier models for the leaderboard (suggested: Groq
  Llama, GPT-4o-mini, Claude Haiku, Gemini Flash).

## Success Criteria

"Done" for v1 means all of:
- A live deployed URL (the playground + leaderboard).
- A public GitHub repo with a strong README: problem, method, results, limitations, cost.
- A reproducible leaderboard covering at least 3 models for hidden-content injection.
- A measured defense result (attack-success drop, ON vs. OFF) with the defense shipped as a
  reusable module.
- The LLM-judge's accuracy reported against a hand-labeled set.
- A 60-90 second demo video of the live hijack, embedded on the portfolio site.

## Distribution Plan

- Primary: web service deployed on Vercel (reuses REVO's pipeline) — existing deployment
  pipeline covers this.
- Code + attack corpus public on GitHub with a strong README.
- Stretch (v2 only): publish the defense module as a small npm package; publish the attack
  corpus as a citable dataset.
- The demo video lives on the builder's portfolio site as the case study.

## Next Steps

Build in this order so there is always a finished, showable slice:
1. **Skeleton:** one cheap model adapter (Groq Llama) + the playground UI summarizing a
   page, clean vs. poisoned, with a canary-leak detector. Proves the core end to end.
2. **Corpus + harness:** author attack corpus v1 (~30-50 payloads across techniques), build
   the harness + LLM-judge, and validate judge accuracy on a hand-labeled subset.
3. **Leaderboard:** add 3-4 models, build the leaderboard page and the methodology writeup;
   cache results.
4. **Defense:** build the mitigation module, run before/after on the corpus, report the
   attack-success drop.
5. **Polish:** README, deploy, record the demo video, write the portfolio case study.

## Engineering Review (2026-06-17, /plan-eng-review)

All decisions below are locked.

### Locked decisions
1. **Compute model:** the leaderboard is PRECOMPUTED by an offline script and committed as a
   results JSON; the site renders it statically. Only the single-attack playground calls
   models live, within Vercel's 60s budget.
2. **Harness engine:** hybrid — wrap promptfoo's TypeScript `evaluate()` to run attacks across
   models; hand-write the indirect-injection layer (poisoned-content builder, canary detector,
   judge, defense module).
3. **Eval scope:** (a) judge accuracy vs a hand-labeled set; (b) defense attack-success ON vs
   OFF (the headline number).
4. **v1 size:** 2 contrasting models + ~25 attacks (defense on/off). Expand in v2. Keeps the
   build solvent on ~$100 and the judge-validation set dense.
5. **Victim scope:** controlled harness (own realistic RAG/summarizer victim, real models via
   official APIs) PLUS one responsible real-world case study — analyze a real public app's
   injection surface from observation / its own docs (no unauthorized attacking), propose a
   defense, and test it in the sandbox.

### Baked-in requirements (from the outside-voice challenge)
- **T1 first:** a 1-hour spike confirming promptfoo's TS lib runs in Vercel Functions from a
  Windows dev box (temp files / bundle size / shell-exec). Pivot the harness if it fails.
- **Sandboxed rendering (SECURITY-CRITICAL):** poisoned content renders ONLY in a sandboxed
  iframe with a locked CSP, or server-side render-to-image. Never inject raw HTML into the page.
- **Staleness:** each leaderboard row shows "tested {date}, model version {x}".
- **Playground:** defaults to curated cached demo pages; live-URL fetch is opt-in and
  time/budget limited.
- **Corpus license:** clear BIPIA (and any reused) licensing in week 1, or author originals and
  state it.
- **Writeup leads with the per-technique bypass breakdown** (which attacks beat which defense).
- **Judge labels:** scale to the 2-model x ~25-attack matrix; keep enough labels per technique
  that accuracy is trustworthy.

### What already exists (reuse, don't rebuild)
- REVO: web-content fetching, Vercel deploy pipeline, React + Tailwind UI. Reuse for the
  playground fetch + UI.
- promptfoo (TS library): the attack-running engine + built-in model providers. Do NOT hand-roll
  providers.
- Public benchmarks (BIPIA et al.): attack-payload source. Do NOT invent the whole corpus.

### NOT in scope (v1)
- Kind 3 (attacking an AI with real tools/actions) — v2.
- 4-model / 50+ payload breadth — v2.
- npm package for the defense; citable dataset for the corpus — v2.
- Running garak/PyRIT (Python) to cross-check — future work, noted in README.
- Unauthorized attacks on live third-party products — out of bounds (the case study is
  observation + sandbox only).

### Failure modes
| Codepath | Failure | Test | Error handling | User sees |
|---|---|---|---|---|
| sanitize.tagUntrusted | payload contains `</untrusted>` and breaks out | required | escape/strip delimiter | blocked |
| judge LLM path | judge gets injected by graded content | required (eval) | hardened prompt + stripped input | n/a |
| playground render | poisoned HTML runs as XSS | required | sandboxed iframe + CSP | safe render |
| playground api | fetch + model exceeds 60s | required | streaming/early-abort; cached pages default | clear timeout msg |
| model adapter | API timeout / rate limit | required | retry + clear error | "timed out, retry" |
| corpus loader | malformed payload JSON | required | schema validation | build-time failure |
| leaderboard | stale results shown as current | n/a | per-row tested-date + version | dated label |

Critical-gap watch: the XSS render path and the judge-injection path must each have a test AND
error handling, or they are silent security failures.

### Worktree parallelization
After T1 (spike) and the shared core (corpus schema + sanitize + canary + judge), two lanes run
in parallel:
- Lane A: harness + leaderboard precompute script (`harness/`, `scripts/`).
- Lane B: playground UI + sandboxed render + api route (`app/`, `api/`).
Both depend on the core. The real-world case study is independent (Lane C, mostly writeup) once
the defense module exists.

### Design tokens (CSS variables)
- Color: `--bg #0B0E14`, `--surface #11151C`, `--border #232A33`, `--text #E6EDF3`,
  `--muted #8B949E`, `--accent #56D4DD` (electric cyan = brand/interactive),
  `--danger #F85149` (attack landed), `--safe #3FB950` (attack blocked), `--warn #D29922`.
- Type: **Geist** (UI) + **JetBrains Mono** (payloads/canaries/code). No Inter or system
  default stacks.
- Spacing: 4px base (4/8/12/16/24/32/48/64). Body text >=16px. Contrast >=4.5:1.
- Semantic color language: red = vulnerable/hijacked, green = defended/blocked, cyan = brand.

### Information architecture (homepage = Hook + Leaderboard)
1. Hero: brand + ONE headline number ("Hidden-text injection succeeds X% of the time. My
   defense cuts it to Y%."), one supporting sentence, one CTA ("Try the playground").
   Tasteful reveal animation allowed: invisible text fading into visible.
2. Leaderboard immediately below: ranked data table (not cards), per-row "tested {date},
   model {version}".
3. Playground one click away. Methodology + real-world case study linked lower.

### Interaction states
| Feature | Loading | Empty | Error | Success | Partial |
|---|---|---|---|---|---|
| Leaderboard | skeleton rows | "results pending, last run {date}" | static fallback + note | ranked table | "3/4 models, Gemini pending" |
| Playground | "Sending to {model}..." spinner on response pane | primed example loaded + "Run attack" CTA | clear timeout/rate-limit msg + Retry | clean vs poisoned panes + verdict chip + canary indicator | streamed response |

Verdict chip: ✓ BLOCKED (green) / ✕ HIJACKED (red) — icon + text, never color alone.

### Responsive
- Leaderboard table -> per-model stacked cards on mobile (the one place cards are right).
- Playground two-pane -> Clean | Poisoned tabs on mobile.

### Accessibility (non-negotiable for a security tool)
- BLOCKED/HIJACKED and clean/poisoned use icon + text label, not color alone (color-blind safe).
- Keyboard nav on all playground controls; visible focus rings.
- ARIA landmarks (nav/main/region); leaderboard table uses proper header scope.
- Touch targets >=44px; body text >=16px; contrast >=4.5:1 verified on the dark palette.

### Anti-slop guardrails
No 3-column feature-card grid; no purple/indigo gradients; no centered-everything; no
emoji-as-design; no icon-in-colored-circle. Leaderboard is a table, playground is a diff —
not card mosaics.
